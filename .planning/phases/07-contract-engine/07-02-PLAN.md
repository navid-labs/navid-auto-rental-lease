    1. Create `src/features/contracts/actions/create-contract.ts`:
       - 'use server' directive
       - createContract(data: { vehicleId, contractType, periodMonths, deposit }) server action
       - Authenticate with getCurrentUser(), require CUSTOMER role
       - Use prisma.$transaction for atomic execution:
         a. Fetch vehicle (include trim.generation.carModel.brand, dealer relation, images) and verify status === 'AVAILABLE'
         b. Check no existing non-terminal contracts (status NOT IN ['CANCELED', 'COMPLETED']) on this vehicle across both RentalContract and LeaseContract tables
         c. Update vehicle status to 'RESERVED'
         d. Calculate payment using calculateRental() or calculateLease() from finance library based on contractType
         e. For LEASE: fetch residualRate from ResidualValueRate table (fallback 0.4 if not found, per Phase 6 convention)
         f. Create contract record (RentalContract or LeaseContract) with status DRAFT, calculated monthlyPayment, totalAmount, deposit
       - Return { contractId, contractType } on success, { error: string } on failure
       - Korean error messages: '이 차량은 현재 계약 신청이 불가합니다.', '이 차량에 대한 진행 중인 계약이 있습니다.'

    2. Create `src/features/contracts/actions/submit-ekyc.ts`:
       - 'use server' directive
       - submitEkyc(data: { contractId, contractType, ekycData: EkycInput }) server action
       - Authenticate, verify contract ownership (customerId === currentUser.id)
       - Call mockVerifyIdentity() -- catch error and return { error: message }
       - On success: create EkycVerification record in DB, update contract status from DRAFT to PENDING_EKYC (or PENDING_EKYC to PENDING_APPROVAL if eKYC was the blocking step)
       - Actually, flow is: step 2 creates contract as DRAFT -> step 3 eKYC verifies -> transitions to PENDING_APPROVAL (skipping PENDING_EKYC as intermediate)
       - Correction per state machine: DRAFT -> step 3 triggers eKYC -> on eKYC success, transition to PENDING_EKYC first, then immediately to PENDING_APPROVAL. Or simpler: treat eKYC submission as the trigger to go DRAFT -> PENDING_APPROVAL (combined).
       - Decision: DRAFT -> PENDING_EKYC when user enters step 3 (auto-transition), then PENDING_EKYC -> PENDING_APPROVAL on successful verification. This preserves the state machine semantics.
       - So: two transitions in submitEkyc: (1) if still DRAFT, first transition to PENDING_EKYC, then (2) on verification success, transition to PENDING_APPROVAL.
       - Return { success: true } or { error: string }

    3. Create `src/features/contracts/actions/update-contract-status.ts`:
       - 'use server' directive
       - updateContractStatus(contractId, contractType, newStatus, role) server action
       - Generic status transition helper used by other actions and admin
       - Validates transition using canTransitionContract()
       - Updates contract record
       - Return { success: true } or { error: string }

    4. Write `create-contract.test.ts` to test:
       - Successful contract creation returns contractId
       - Rejects when vehicle not AVAILABLE
       - Rejects when existing active contract exists (double-booking)
       - Rejects unauthenticated users
       - Uses vi.hoisted() for Prisma mock
